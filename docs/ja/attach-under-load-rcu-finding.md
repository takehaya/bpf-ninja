# 高負荷トラフィック中の attach がハングする機序と修正

高負荷の XDP データパスに対して xdp-ninja を起動すると、1 パケットも捕捉できないことがあった。
本書はその原因を突き止めた経緯と機序、修正内容、修正後の測定結果を記録する。

計測環境は次のとおり。
64 論理 CPU のホスト（kernel 6.15.0-rc6 カスタムビルド、Intel E810）で、cpumap により全コアへパケットを分散する XDP プログラムを対象に、トラフィックジェネレータで高 pps を印加した。

## 症状

流量を段階的に上げながら、その都度 xdp-ninja を起動して数秒間 pcap 捕捉した。
結果は次のようになった。

| offered (総流量) | 捕捉 |
|---|---|
| 19 kpps | ほぼ 100% |
| 95 kpps | ほぼ 100% |
| 284 kpps | ほぼ 100% |
| 662 kpps | ほぼ 100% |
| 1.14 Mpps | ほぼ 100% |
| 1.42 Mpps | **0** |
| 1.70 Mpps | **0** |

ある流量を境に、部分的な取りこぼしではなく捕捉が完全にゼロへ落ちる。
`--arg-filter` で対象を全体の 0.1% に絞っても、`--fast-reader` や `--rx-cores` でリーダーを別コアへ逃がしても、ゼロのままだった。

この振る舞いは「捕捉スループットの上限を超えた」という説明と合わない。
上限超過なら、超過分だけ取りこぼして残りは捕捉されるはずである。
崖のように 100% から 0 へ落ちるのは、連続量の飽和ではなく、何かが離散的に壊れていることを示す。

## 切り分け

手がかりは二つあった。

第一に、`--arg-echo`（引数診断モード）は同じ負荷でも正常に動いた。
arg-echo は捕捉用の per-CPU シャード ringbuf を使わず、専用の単一 ringbuf を 1 個だけ作る。
つまり fentry attach 自体やフィルタ評価は生きており、壊れているのはシャード ringbuf 経路のどこかに絞られる。

第二に、高負荷中の実行では捕捉ゼロどころか、起動直後に出るはずの `capturing ...` 行すら表示されないことがあった。
捕捉ループに到達していない。
つまりこれは捕捉の問題ではなく、**セットアップ（マップ作成から attach まで）の問題**である。

そこで高負荷（1.7 Mpps）中に xdp-ninja を起動し、ハングした時点のカーネルスタックを `/proc/<pid>/task/*/stack` で取得した。

```
[<0>] __wait_rcu_gp+0x18f/0x1a0
[<0>] synchronize_rcu_normal.part.0+0x47/0x70
[<0>] synchronize_rcu+0xef/0x110
[<0>] map_update_elem+0x30d/0x350
[<0>] __sys_bpf+0xf5d/0x2710
```

`map_update_elem` の中で `synchronize_rcu` を待っている。
ハング箇所はマップ更新である。

## 機序

xdp-ninja の捕捉経路は、CPU ごとに 1 個の RingBuf を用意し、それらを 1 個の ARRAY_OF_MAPS（outer map）に格納する構造をとる（`createShardedRingbuf`）。
BPF 側は自分の CPU 番号で outer を引き、自 CPU のシャードに書く。

この outer への格納が問題だった。
ARRAY_OF_MAPS と HASH_OF_MAPS の要素更新に限り、カーネルは syscall が返る前に `synchronize_rcu()` を呼ぶ（`kernel/bpf/syscall.c` の `maybe_wait_bpf_programs()`）。
これは inner の解放を守るためではない（解放自体は参照カウントと RCU コールバックで別途守られている）。
「update が返った時点で、古い inner を使って走行中の BPF プログラムはもういない」という保証をユーザ空間に与えるためで、map-in-map の典型ユース（新しい inner へ差し替えてから古い inner の中身を回収する）を安全にする仕様である。
BPF プログラムは RCU の read-side critical section 内で走るため、この保証は RCU の **grace period**（差し替え時点で走行中だった reader がすべて抜けるまでの待ち。実装上は全 CPU が静止状態を通過するまで）を 1 回待つことで得られる。
つまり更新 1 回につき grace period 1 回を、無条件で待つ。

無負荷なら grace period はミリ秒未満で終わり、64 回繰り返しても気にならない。
ところが grace period は全 CPU の静止状態が揃うまで閉じないため、NAPI ポーリングが softirq を回し続けて静止状態を通す隙の減った CPU が 1 つでもあると、全体の待ちが伸びる。
この環境では、XDP 負荷をかけると 1 回あたりおよそ 1.5 秒まで伸びた（修正後のセットアップ所要時間からの逆算）。
そして従来の実装は 64 個の inner を **逐次** Put していたため、セットアップ全体では grace period を 64 回直列に待つ。
1 回 1.5 秒なら合計はおよそ 96 秒であり、数秒で SIGINT を受ける実運用では「永久にハングしている」ように見える。

捕捉ゼロに見えた理由もここにある。
SIGINT ハンドラは捕捉ループ開始時（`capturing ...` 表示の直後）に登録されるため、セットアップ中に SIGINT を受けるとプロセスは既定動作でそのまま終了する。
サマリ行も出力されず、外からは「起動したのに 1 パケットも撮れなかった」ようにしか見えない。

先の表の「崖」は、この機序で説明できる。
崖の位置（この構成では 1.1 Mpps と 1.42 Mpps の間）は捕捉能力の限界ではなく、**grace period が測定ウィンドウ内にセットアップを終えられなくなる負荷の閾値**である。
実際、対象の XDP プログラムを少ないコアに寄せて softirq の飽和を緩めると、崖の位置は 1.42 Mpps の先へ動いた。
また、無負荷時に attach を済ませてから同じ 1.70 Mpps を印加すると、15 秒間で 2,100 万パケットを捕捉できた。
読み取り経路そのものは、この流量を最初から処理できていた。

## 修正

inner map の作成はもともと並列化されていた（syscall コスト対策）。
一方で outer への Put だけが逐次だった。
Put も並列化する。

```go
// 修正前: 逐次 Put。1 回ごとに synchronize_rcu を待つ
for i, inner := range inners {
    if err := outer.Put(uint32(i), inner); err != nil { ... }
}

// 修正後: 並列 Put。grace period が合流し、待ちは実質 1 回分になる
for i, inner := range inners {
    go func(i int, inner *ebpf.Map) { putErrs[i] = outer.Put(uint32(i), inner) }(i, inner)
}
```

並列に発行された `synchronize_rcu` は同じ grace period を共有できるため、64 回分の待ちがおよそ 1 回分に潰れる。

## 修正後の測定

1.70 Mpps を流し続けた状態で、修正版をコールドスタートした。

| | 修正前 | 修正後 |
|---|---|---|
| セットアップ完了まで | 25 秒以上（未完了） | **1.5 秒** |
| 捕捉 | 0 | **19.5M パケット（約 13 秒間、実質 1.5 Mpps）** |

崖は消え、負荷中の起動でも数秒で捕捉が始まる。

## 残る制約

この修正で直るのはセットアップだけである。
測定の過程で見つかった別の制約を二つ記録しておく。

**pcap 書き込み経路の上限**：offered 1.42 Mpps に対し、`--null-output` では約 1.2 Mpps 以上を読み切れたが、`-w` での pcap-ng 書き込みは 1.1 Mpps 前後で頭打ちになった。
律速は読み取りではなく書き込み側にある。
要因の分解（ファイルシステム、ページキャッシュ、シリアライズ）は未実施。
高レートでの取得には `--raw-dump`（生バイト書き出しとオフライン変換）が現実的な回避策になる。

**producer コア飽和時のリーダー枯渇**：ラインレート近く（この環境で 78 Mpps）まで印加すると、事前に attach してあっても捕捉はゼロになった。
シャードのリーダーは既定で producer と同じ CPU に pin されるため、その CPU が softirq で飽和するとユーザ空間のリーダーが走れなくなる、という機構だと考えている。
attach ハングとは別の問題であり、こちらは未修正。
対象トラフィックが通るコアに余裕がある運用（本来のユースケースである少数フローの狙い撃ち）では顕在化しない。

## 再現と診断の手順

同種の症状（高負荷中だけ捕捉ゼロ、または起動が返ってこない）を疑うときは、次の順で確認するとよい。

1. `capturing ...` 行が出ているかを見る。出ていなければセットアップで止まっている。
2. ハング中に `sudo cat /proc/<pid>/task/*/stack` を取り、`synchronize_rcu` を含むスタックを探す。
3. 無負荷時に attach してから負荷を印加し、捕捉できるかを比べる。捕捉できるなら本書と同じ機序である。
